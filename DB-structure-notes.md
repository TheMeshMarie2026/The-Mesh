To keep your development lightning-fast for a 48-hour hackathon, use a lightweight relational schema or a flexible document-store database (like Supabase/PostgreSQL or Firebase Firestore).
The goal is a highly normalized, high-performance database schema that tracks the evolving game state, the shifting node priority metrics, and the real-time node chat dialogue generated by n8n.

# Entity-Relationship Overview
 [GameSession] 1 ------ * [GameTurn] 1 ------ * [TurnNodeState]
                     1 ------ * [NetworkChatLog]

1. GameSession Table (Global Status)
Tracks the active players, the overall health of the environment, and whether the game is won or lost.

|Column Name	|Data Type	|Constraints / Description
|-------|-----|-------|
|session_id	|UUID	|Primary Key
|player_count	|Integer	|Number of active participants (humans + AI bots)
|ecosystem_health	|Float	|Current health of the environment (0.0 to 100.0)
|current_turn	|Integer	|Track progression (1 through 10)
|game_status	|Enum	|['ACTIVE', 'VICTORY', 'COLLAPSE']
|created_at	|Timestamp	|Session initiation marker

2. GameTurn Table (Crisis Definition)
Stores the unique attributes of the threat spawned at the start of each turn.
Column Name	Data Type	Constraints / Description
turn_id	UUID	Primary Key
session_id	UUID	Foreign Key referencing GameSession(session_id)
turn_number	Integer	1 to 10
crisis_name	String	E.g., "Chemical Gas Leak"
crisis_x	Float	Grid location coordinate X for distance calculation
crisis_y	Float	Grid location coordinate Y for distance calculation
demand_medical	Float	Total medical resource required (0.0 to 1.0+)
demand_logistics	Float	Total logistical resource required (0.0 to 1.0+)
demand_water	Float	Total water resource required (0.0 to 1.0+)
resolution_status	Enum	['PENDING', 'SUCCESS', 'FAILURE']

3. TurnNodeState Table (The Heterarchy Dynamic)
This is the most critical table. It logs how individual node vectors and priority scores dynamically mutate contextually turn-by-turn.
Column Name	Data Type	Constraints / Description
state_id	UUID	Primary Key
turn_id	UUID	Foreign Key referencing GameTurn(turn_id)
node_id	String	Unique node identifier (e.g., "AI_Medic_Agent")
is_ai	Boolean	True for agent, False for human player
node_x	Float	Grid coordinate X (to calculate Euclidean distance)
node_y	Float	Grid coordinate Y (to calculate Euclidean distance)
priority_score	Float	The contextual value generated by your Python Engine
network_rank	Integer	The calculated power rank for this turn (1 to 5)
network_stance	Enum	['LEADERSHIP', 'DEFERRAL']
target_node	String	If deferring, who they route line vectors to
contrib_medical	Float	The node's raw medical contribution this turn
contrib_logistics	Float	The node's raw logistical contribution this turn
contrib_water	Float	The node's raw water contribution this turn

4. NetworkChatLog Table (n8n LLM Output Storage)
Stores the immersive dialogue payloads returned by your n8n workflow so your Lovable front-end can poll or listen to them in real-time.
Column Name	Data Type	Constraints / Description
chat_id	UUID	Primary Key
turn_id	UUID	Foreign Key referencing GameTurn(turn_id)
node_id	String	The node speaking
dialogue_text	Text	The 1-3 sentences generated by the LLM Schema
timestamp	Timestamp	Sequence order enforcement marker

Hackathon Implementation Strategy (Supabase Integration)
If you use Supabase, you can write a PostgreSQL Database Function (RPC) to execute your math engine straight inside the database using SQL, rather than needing a separate Python server.
When a user submits their turn choices, your front-end calls the database function. The database updates the health pool, calculates the new rankings, and triggers an n8n webhook via an automated Database Webhook Event to prompt the LLM strings seamlessly.
